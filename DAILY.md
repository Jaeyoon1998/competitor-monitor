# 데일리 루틴 지침

이 문서는 매일 실행되는 에이전트가 그대로 따르는 절차다. 사람이 읽어도 되지만,
1차 독자는 스케줄된 Claude Code 세션이다.

작업 디렉터리: `C:\Users\youjy\competitor-monitor`

---

## 1. 수집

```
node collect.js
```

- 출력: `data/raw/YYYY-MM-DD.json` — **이번 실행에서 처음 본 항목만** 들어 있다.
- `items` 가 비어 있으면 신규 신호가 없다는 뜻이다. 2~4단계를 건너뛰고 5단계로 간다.
- `errors` 배열을 반드시 확인한다. 수집기 실패는 조용히 넘어가지 말고 5단계 보고에 포함한다.
  - `play_release` 실패가 이틀 연속이면 구글이 페이지 구조를 바꾼 것이다.
    `collect.js` 의 `findRecentChanges` 경로 상수를 갱신해야 한다.

## 2. 분류

`data/raw/<오늘>.json` 의 각 항목을 읽고, **경쟁 대응에 의미 있는 것만** 이벤트로 승격한다.

이벤트로 승격할 것:
- 기간이 정해진 인게임 이벤트, 부스트, 출석/미션
- 유료 상품 출시·할인·패스·스킨 판매
- 콜라보 / IP 제휴
- 신규 콘텐츠나 시스템이 들어간 업데이트
- 대회, 개발자 방송, 대형 크리에이터 캠페인

승격하지 말 것:
- 버그 수정만 있는 패치 노트
- 일상적인 커뮤니티 잡담, 밈 영상
- 이미 등록된 이벤트의 후속 홍보물 (기존 이벤트의 `evidence` 에 URL만 추가한다)

유형(`type`)은 다섯 개뿐이다. 새로 만들지 않는다.

| type | 뜻 |
|---|---|
| `promo` | 과금 프로모션 — 유료 상품, 할인, 패스, 스킨 |
| `event` | 인게임 이벤트 — 기간제 콘텐츠, 부스트, 미션 |
| `update` | 업데이트 — 신규 콘텐츠·시스템 |
| `collab` | 콜라보 / IP 제휴 |
| `marketing` | 마케팅·미디어 — 대회, 방송, 크리에이터 |

## 3. `data/events.json` 갱신

`events` 배열에 항목을 **추가**한다. 스키마는 파일 상단 `_schema` 참고.

- `id`: `<competitor_id>-<starts_on>-<영문 슬러그>`
- `starts_on` / `ends_on`: 공지에 기간이 명시돼 있으면 그대로. 없으면 `ends_on: null`
  (대시보드가 "진행 중"으로 처리하고 막대 오른쪽을 페이드 처리한다).
- `confidence`: 기간이 명시돼 있으면 `high`, 추정이면 `medium`, 정황뿐이면 `low`.
- `summary`: **무엇을 / 언제 / 어떤 보상으로** 를 1~2문장. 원문 복붙 금지.
  가능하면 "왜 이렇게 했는가"에 대한 해석을 한 문장 붙인다.
- `signals`: 대응 판단에 쓸 키워드 2~4개.
- **기존 항목 수정 규칙**: 이미 있는 이벤트의 종료일이 새로 밝혀지면 `ends_on` 을 채운다.
  이벤트를 삭제하지 않는다 — 종료된 이벤트도 히스토리로 남긴다.

## 4. 빌드

```
node build-dashboard.js
```

검증에 실패하면(필수 필드 누락, 알 수 없는 type, 중복 id 등) 오류를 고치고 다시 빌드한다.

그다음 `dashboard.html` 을 Artifact 로 게시한다. **반드시 같은 URL 로 갱신한다** —
`url` 파라미터에 아래 값을 넣는다:

```
ARTIFACT_URL: https://claude.ai/code/artifact/8324f21b-2e54-45a7-95ef-cfd59c1fbf9d
```

## 5. 보고

Slack 이나 요약 메시지로 다음을 전달한다:

- 오늘 신규 감지 N건 — 경쟁사별 한 줄씩
- 그중 **주목할 것** 1~3개와 그 이유 (특히 `promo` 유형, 그리고 우리 일정과 겹치는 건)

신규가 0건이면 "신규 없음" 한 줄이면 된다. 억지로 분량을 채우지 않는다.

## 주의

- 자동 수집이 닿지 않는 곳: **Meta/TikTok 광고 라이브러리, 인게임 배너, 로그인이 필요한 카페**.
  공개 API가 없다. 주 1회 사람이 직접 확인하고 events.json 에 손으로 추가하는 것을 전제로 한다.
- `appstore_iap` 항목은 **가격이 바뀌었을 때만** 나온다. 나오면 거의 항상 `promo` 이벤트감이다.
  본문에 `[추가]` / `[삭제]` diff 가 들어 있으니 무엇이 얼마에서 얼마로 바뀌었는지 요약에 적는다.
  동일 수량 상품이 두 가격으로 동시 노출되면 할인 프로모션이거나 가격 테스트다.
- `data/seen.json` 은 중복 방지 지문이다. 지우면 다음 실행에서 전부 신규로 잡힌다.
  `data/iap-state.json` 은 인앱 가격 비교 기준선이다. 지우면 다음 실행에서 diff 없이 전량 재수집된다.
- 수집 실패 하나로 루틴 전체를 멈추지 않는다. 되는 것까지 처리하고 실패를 보고한다.
